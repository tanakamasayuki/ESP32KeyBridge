# Core Design

`ESP32KeyBridge` core は、特定の入出力方式に依存しないキーボード系入力の統合・変換エンジンです。中間データ表現の定義は [DATA_MODEL.ja.md](DATA_MODEL.ja.md) にあります。この文書はパイプラインとモジュール境界を定義します。

USB Host / USB Device、GPIO、BLE、UART、WebSerial、NVS、JSON などは core の外側に置きます。必要な依存はアダプタまたは example が持ちます。

## データフロー

```text
物理入力
  -> 入力アダプタ       キーイベント / 文字列 / 相対値を生成。入力ごとの押下集合
  -> 入力ごとの変換     per-input remap・無効化、(opt-in)レイアウト変換
  -> 統合               押下集合は和集合、単発イベントは到着順で合流
  -> 全体変換           global remap / レイヤー / マクロ / 無効化
  -> 出力の縁           文字ストリームの打鍵展開、アクション列キュー、相対値の合算
  -> 出力アダプタ       表現できるものを report 化・送信

逆方向:
  出力アダプタ(lock 報告)-> 内部 LockState シャドウ -> 全入力アダプタ(LED 等へ反映)
```

- per-input と global の変換は意味が異なります。per-input は「この入力デバイスから来たキーだけ」、global は「どの入力から来ても統合後に」。レイアウト変換は刻印が入力デバイスの属性なので per-input に置きます。
- レイヤー状態・マクロは統合後レベルで動作し、既定で全入力に効きます。

## 入力アダプタの責務

- raw 入力(USB report、BLE、GPIO スキャン、シリアル)を読み、キーイベント / 文字列 / 相対値へ変換する。keyboard のキーは raw HID usage のまま入れる(**入力アダプタは layout 解釈をしない**)。
- スキャン、デバウンス、座標→キー割り当て(マトリクス)、ペアリング(BLE)、プロトコル解釈など、ハードウェア・トランスポート固有の処理を閉じ込める。
- **時間依存の処理(デバウンス、タイムアウト、リピート判定)はアダプタ側**。core は時刻を持たない。
- 論理押下の合成(トグル、ワンショット、エンコーダのタップ化)。
- 切断・消失の検出(検出したら core が押下集合をクリアする)。
- core からの LockState 通知を受け、LED report / GPIO ピン等へ反映する(任意実装)。

## 出力アダプタの責務

- 統合・変換済みの状態と単発イベントを、自分が表現できる範囲で report 化・送信する(表現できないものは黙って捨てる)。
- アクション列キューを自分の送信レート(HID poll 間隔)で消化する。修飾変更と打鍵は別 report に分ける。
- USB device / BLE HID アダプタは、ホストからの LED output report を受信して core へ lock 状態を報告する(lock 報告出力)。
- USB descriptor(複合構成を含む)、プロトコル(boot/report)、送信タイミングはアダプタの領分。構成は enumerate 時に確定する。

## 時間・並行性の境界

時間・割り込み・タスクは**アダプタの内側に閉じ込め**、core と `update()` の呼び出しコンテキストは単一のまま保ちます。

- サンプリングの時間精度(マトリクスのスキャン周期、デバウンス、エンコーダのエッジ取りこぼし防止)は、アダプタ内部の task / ISR / ペリフェラルが担保する。想定実装: GPIO マトリクスは専用 task か周期タイマで約 1kHz スキャン、ロータリーエンコーダは PCNT(パルスカウンタ)ペリフェラルでハードウェア計数し `update()` でカウンタ差分を読む(ISR 不要)。
- USB Host / USB Device / BLE はスタック側の task が既に存在する。アダプタはコールバックで受けたイベントを内部キューへ積む。
- アダプタの責務は「内部の並行処理から `update()` 呼び出しコンテキストへのスレッドセーフな引き渡し」まで。押下集合への反映・変換・出力はすべて `update()` の単一コンテキストで行う。
- この分担により、スケッチの loop が他処理で詰まっても入力の取りこぼしは起きない(アダプタ内で確定済み)。遅れるのは変換と出力だけ。
- ブリッジ自身を task で自動駆動する仕組みは core には入れない(排他が `applyConfig()` / `typeText()` へ波及するため)。必要になれば Arduino 層の opt-in ラッパーとして追加する([DECISIONS.ja.md](DECISIONS.ja.md) 将来メモ)。

## 制御プレーン

ブリッジはデータの流路とは別に、少数の**モード状態**を持ちます。

- レイヤーの有効状態(virtual キーの押下で変化)
- レイアウト変換の on/off(ホットキー = virtual キー、または外部コマンドで変化)
- (将来)アクティブ出力の選択

virtual キーの消費先は「レイヤー」「マクロ起動」「モード変更」の 3 系統です。外部からの設定変更(シリアル CLI、WebSerial 等)も同じ制御プレーンに入ります。設定変更が適用された瞬間に押下中のキーがある場合も、press 時確定の原則により release は破綻しません。

## 依存境界と移植性

core は移植可能な純粋 C++ として実装し、プラットフォーム依存はすべてアダプタ層に置きます。アダプタ層が実質的な HAL です。

core が依存してよいもの:

- C++ 標準(C++17 まで)の言語機能と `<stdint.h>` / `<stddef.h>`。
- それ以外なし。Arduino API、ESP-IDF、FreeRTOS、`millis()`、`PROGMEM` などプラットフォーム固有のものには依存しない。

core が守るルール:

- **時刻を取得しない。** タイムスタンプが必要な場合はアダプタ(または呼び出し側)が付与する。時間依存処理はアダプタ側か、`update()` への時刻注入で行う。
- **動的メモリ確保をしない。** すべて固定容量で構成する。例外・RTTI も使わない。
- **単一コンテキスト駆動。** core は `update()` を呼ぶ 1 つの実行コンテキストで動作する前提とし、割り込みや別タスクからのイベント投入の直列化はアダプタ側の責務にする。
- レイアウト記述などの定数データは `const`(flash / rodata 配置)とし、プラットフォーム固有のアクセス方法を前提にしない。

この境界により、core はホスト PC 上でそのままコンパイル・テストでき(unit test はこの性質で成立)、ESP32 以外のプラットフォームへもアダプタを書くだけで移植できます。SoC ごとの機能差(USB OTG の有無、BLE の有無)はアダプタの提供有無として現れ、core には現れません。

## Configuration Boundary

core は設定 object を受け取って適用します。

core が規定しないもの:

- 設定をどこから取得するか、どこへ保存するか。
- JSON、binary、C++ builder などの外部形式。
- WebSerial、UART、BLE、TCP などの転送手段。設定画面 UI。

core が持つもの:

- C++ API で設定を構築する入口、検証する入口、適用する入口。

設定に含まれるもの: remap 表(種別またぎ)、無効化、レイヤー定義、マクロ(キー列 / 文字列)、per-input のレイアウト変換指定、出力のホストレイアウト指定、相対値軸の反転・倍率。
