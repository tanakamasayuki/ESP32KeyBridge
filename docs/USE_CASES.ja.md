# ユースケース検証

代表的なユースケースを仕様([DATA_MODEL.ja.md](DATA_MODEL.ja.md))に当てて検証した記録です。各ユースケースの検証から仕様へ反映した決定は [DECISIONS.ja.md](DECISIONS.ja.md) にまとめています。

## UC1: Ctrl と CapsLock の入れ替え

キーボードと PC の間に入れ、キーボード側・OS 側の設定を変更せずに 2 キーを入れ替える。

- **実現**: global remap 2 エントリ(`0x39→0xE0`、`0xE0→0x39`)。remap は 1 段 lookup で連鎖しないため 2 行で入れ替えになる。
- **性質**: layout・OS 設定・刻印に非依存。長押し・同時押し・リピート・入れ替え後の Ctrl+C まで完全に保たれる。
- **LED**: 入れ替え後の Ctrl 位置で Caps が toggle しても、ホストからの LED report を転送するので物理キーボードの LED は正しく追随する。

## UC2: 複数キーボードの状態共有(分割キーボード)

片側の Shift を反対側のキーに効かせる。

- **実現**: 入力ごとの押下集合の和集合統合のみ。修飾キーを「ただのキー」として扱う正規化により、共有のための追加機構はゼロ。
- **補足**: OS も修飾キーはグローバル共有するため、PC 直挿しでも同じ挙動になる。ブリッジ統合が本質的に必要なのは、出力先に OS がいない構成と、レイヤー等のブリッジ内機能を入力横断で効かせる場合。

## UC3: GPIO マトリクスで自作キーボード(LED 付き)

- **実現**: スキャン・デバウンス・座標→キー割り当てはすべて入力アダプタ側。core からは USB キーボードと同型の入力に見える。
- **LED**: LockState の通知を GPIO ピン出力として実装する(通知手段はアダプタ依存)。
- **制約**: マトリクス規模は GPIO 本数(IO エキスパンダで拡張可)、ゴースト対策はアダプタのスキャンロジックの領分。

## UC4: BLE キーボードを有線(USB)キーボードに変換

- **実現**: BLE 入力アダプタ(HID over GATT、ペアリング等を内包)+ USB device 出力。変換なしの素通し。
- **ここで確定した規則**: **入力の切断・消失時は、その入力の押下集合をクリアして全 release 扱いにする**(修飾キー押下中の電池切れで stuck key にならないため)。BLE 固有ではなく USB 抜去等にも適用される一般規則。

## UC5: US 刻印キーボードを日本語 Windows で使う(レイアウト変換)

OS 設定を変えずに刻印どおり打てるようにする。

- **実現**: per-input のレイアウト変換((キー,Shift)→(キー,Shift) 表 + Shift 抑制・合成)。英字・数字単打・ナビゲーションは恒等で素通し、変換対象は記号系のみ。IME キー(変換/無変換等)の供給は通常の remap。
- **実用性**: ソフトウェア実装(ULE4JIS 等)と QMK の同手法(keymap_jp.h)で実証済みのカテゴリ。US→ja_jp 方向は ASCII 被覆 100%。
- **必須要件**: 変換 on/off の切替手段(BIOS 等、別解釈環境での化け対策)。
- **制約**: 記号入りショートカットの二義性(既定は修飾コンボ素通し)、Shift 要求が矛盾する同時押しは非対応。

## UC6: US 配列バーコードリーダーを日本語 Windows で使う

- **実現**: HID 型リーダーは UC5 と同じ変換表。シリアル型リーダーは文字ストリーム入力として扱い、出力の縁で打鍵展開。
- **好条件**: 機械タイピストはショートカット・同時押し・ロールを発生させないため、変換の制約がほぼ効かない。
- **制約**: IME オンのフィールドへのスキャンは化ける(不可能性 #4)。本筋は運用・アプリ側対処で、ブリッジは前置キー等の緩和オプションのみ。

## UC7: フットスイッチ・GPIO 単独キー

- **実現**: 1 キーのキーボードとしてアダプタ数行。仮想キーに割り当てれば「踏んでいる間だけレイヤー」が入力横断で効く。
- **ここで確定した規則**: 押下集合は**論理押下**であり物理と 1:1 でなくてよい。トグル・ワンショット・エンコーダのパルス合成はアダプタが論理押下を作る。時間依存の判定はアダプタ側。

## UC8: レイアウト混在の複数キーボード(US + JIS 併用)

- **実現**: per-input 変換(統合前)+ 和集合統合。JIS 側は素通し、US 側だけ変換。**個別に PC へ挿す方式では原理的に解決できない**(OS layout は単一、OS 修飾はグローバル)。
- **制約**: 一方の修飾押下中に他方の変換キーが同時到来した瞬間の干渉のみ。直列化と遅延オプションで緩和。

## UC9: メディアキー(IR リモコン、ロータリーエンコーダ、Fn レイヤー)

- **実現**: Consumer キーは種別 `consumer` の継続系として押下集合に乗る。種別またぎ remap により「F13 → 音量 Up」「リモコンのボタン → PageDown」が書ける。エンコーダはアダプタがタップ合成、IR はリピートフレームで論理押下を復元。
- **確定**: エッジ動作(Play/Pause)か連続動作(音量)かはホストの解釈で、ブリッジは関知しない。

## UC10: 文字列マクロ・外部文字列のタイピング

ペダルで定型文、シリアルで受けた文字列を打鍵(テスト自動化)。

- **実現**: 文字ストリーム(マクロ / 外部指令 / 文字列デバイス共通)→ 出力の縁でキーボードレイアウト記述により打鍵展開 → アクション列キュー。制御文字は標準割り当て(LF/CR→Enter、TAB→Tab 等)。
- **制約**: 打てない文字は読み捨て + カウント。IME オフ前提。他入力とは文字単位の修飾退避/復元で共存(遅延オプションあり)。

## UC11: マウス(remap、BLE⇄USB 変換、統合)

- **実現**: ボタンは継続系(統合・種別またぎ remap・レイヤー・切断クリアを全適用)。移動・ホイールは相対値の単発系(加算圧縮、report 合算 + 繰り越し、軸の反転・倍率)。
- **構成**: マウス report が出るのはマウス対応の出力アダプタ(複合 USB device 等)を登録した場合のみ。descriptor は enumerate 時に確定。
- **対象外**: 絶対座標(タブレット・タッチ)は v1 対象外。
