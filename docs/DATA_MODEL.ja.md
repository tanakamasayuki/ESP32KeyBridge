# 中間データモデル

この文書は、`ESP32KeyBridge` の中間データ表現を定義します。入力アダプタ、変換パイプライン、出力アダプタは、すべてここで定義する表現だけで接続します。

ユースケースごとの検証記録は [USE_CASES.ja.md](USE_CASES.ja.md)、決定の経緯と状態は [DECISIONS.ja.md](DECISIONS.ja.md) を参照してください。

## 設計原則

1. **ブリッジは解釈しない。** キーが何の文字を意味するかの解釈は、原則としてホスト(PC の OS)の仕事です。ブリッジはキーの識別子を運び、変換が必要な場合だけ限定的に解釈します(レイアウト変換・文字列打鍵)。
2. **状態の正本は、その状態を解釈する側が持つ。** ブリッジは解釈しない情報の正本を持ちません(Lock 状態の扱いがこの原則の適用例です)。
3. **時間を持たない。** core は時刻を取得せず、時間依存の処理(デバウンス、ペーシング、タイムアウト)はアダプタ側に閉じます。

## キーの同一性: 種別 + 値

キー・ボタンの同一性は **(種別, 値)** の組で表します。種別ごとに値空間が独立しています。

| 種別 | 値の意味 | 例 |
|------|---------|-----|
| keyboard | HID keyboard page(0x07)の Usage ID と数値恒等 | A=0x04、Enter=0x28、LeftCtrl=0xE0、CapsLock=0x39 |
| consumer | HID consumer page(0x0C)の Usage ID と数値恒等 | 音量Up=0x00E9、Mute=0x00E2、Play/Pause=0x00CD |
| mouse_button | ボタン番号 | 左=1、右=2、中=3 … |
| virtual | ブリッジ内部専用(レイヤートリガ、マクロトリガ、モード切替) | 任意採番。**出力アダプタへは決して出ない** |
| (将来拡張) | System Control、絶対座標系 など | |

- keyboard の値は **layout 非依存の物理キー識別子**です。JIS keyboard も US keyboard も、A の位置のキーは同じ 0x04 を送ります。刻印の解釈はホストが行います。
- HID 由来の種別を Usage と数値恒等に保つことで、パススルーとリマップは**変換テーブルを一切必要としません**。
- データ表現(構造体のペアで持つか、整数の領域分けか)は実装判断とします。仕様として確定するのは「同一性 = 種別 + 値」であることだけです。

## トラフィックの 2 大分類

中間表現を流れるものは、次の 2 種類に分類されます。新しい入力・出力を追加するときは、まずこの分類を判定すれば処理規則が機械的に決まります。

| | 継続(状態系) | 単発(イベント系) |
|---|---|---|
| 意味 | 「押している間」に意味がある | 発生した瞬間がすべて。1 回消費で終わり |
| 表現 | 押下集合(スナップショット) | 順序付きの列(キュー) |
| 統合 | 和集合 | 到着順で合流 |
| 切断時 | クリアして全 release | 未消化分を捨てる |
| リピート | ホストが状態から生成 | 概念がない |
| 例 | キーボードのキー、修飾キー、Consumer キー、マウスボタン | 文字、キー操作指示(マクロ)、マウス移動・ホイール |

- **Consumer キーは継続系**です。エッジで反応するか(Play/Pause)押下中ずっと反応するか(音量)はホストの解釈であり、ブリッジは関知しません(keyboard の Enter と CapsLock の違いと同じ態度)。
- **マウスはボタン = 継続、移動・ホイール = 単発(相対値)**です。
- Lock 状態は「逆方向(ホスト → ブリッジ → 入力)の状態系」です(後述)。

## 継続系(状態)の規則

### 押下集合

- 入力ごとに押下集合を保持し、統合はその**和集合**です。あるキーの release は「全入力がそのキーを離したとき」に成立します。
- 修飾キー(0xE0–0xE7)も通常キーと同じ規則で押下集合に入ります。HID report 上の modifier ビットとキー配列バイトの違いは、入力アダプタが「ただのキー」に正規化し、出力アダプタが report 生成時に振り分けます。
- 押下集合は**論理押下**です。物理状態との 1:1 対応は要求しません。トグル(踏むたびに Shift の press/release を交互発行)、ワンショット、エンコーダのパルス(タップ合成)などは、アダプタが論理押下を合成して表現します。
- **入力が切断・消失したら、その入力の押下集合をクリアし、全 release として統合へ反映します**(stuck key 防止)。切断の検出はアダプタ、クリアと反映は core の責務です。

### リマップ

- リマップは **1 段の lookup で、連鎖しません**。`CapsLock→LeftCtrl` と `LeftCtrl→CapsLock` の 2 エントリで入れ替えが成立します。
- リマップは**種別をまたげます**(例: キーボードの F13 → 音量 Up、マウスのサイドボタン → キーボードキー、任意キー → virtual)。
- キー無効化(捨てる)も同じ枠組みです。

### レイヤー

- 物理 Fn キーは HID に現れないため、観測・共有・remap とも不可能です(原理的制約)。Fn 相当は「HID に飛ぶキーを virtual キーへ remap して消費し、押下中だけレイヤー(重ね置換表)を有効化する」で実現します。自作キーボードはアダプタの割り当てで直接 virtual キーを産めます。
- **置換結果は press 時に確定し、release は press 時の確定値を離します**(レイヤー解除後の release で stuck key を作らないための必須規則)。
- レイヤー状態は統合後レベルで管理し、既定で**全入力に効きます**(分割キーボードやフットペダル Fn の根拠)。
- tap/hold(短押しで元キー、長押しで Fn)は時間依存のため v1 対象外(将来メモ)。

## 単発系(イベント)の規則

単発系の payload は 3 種で、規則(順序保存・到着順合流・キュー)は共通です。

### 文字ストリーム

「この文字列を順番に打ちたい」を表す、一方向の Unicode 文字の列です。

発生源:

- 文字列マクロ(virtual キーのトリガ消費と引き換えに発生)
- 外部指令(シリアル / ネットワークから受けた文字列)
- 文字列デバイス(シリアル接続バーコードリーダー等)

規則:

- 文字列は**文字のまま出力の縁まで運び**、出力側のキーボードレイアウト記述で打鍵列へ展開します(マクロ定義を出力先レイアウトから独立させるため)。
- remap・レイヤーは文字ストリームに効きません(キー専用)。
- **エスケープ解釈はしません。** `"\n"` の解釈は文字列を作る側(コンパイラや parser)の責務で、ブリッジは実際の制御文字だけを見ます。

受け入れる文字種(入力は UTF-8 で任意のコードポイントを受け付け、打鍵化できるかは次で決まる):

| 分類 | 挙動 |
|---|---|
| ホストレイアウト表にある印字文字 | 打鍵化(必要な Shift を合成、CapsLock は内部シャドウで補正、数字は keypad ではなく主キー列) |
| 制御文字 5 種(下表) | 対応キーを打鍵 |
| それ以外(かな漢字・絵文字・表にない記号・その他の制御文字) | 読み捨て + エラーカウント(`textEncodeFailCount`) |

プリセットごとの印字文字の範囲:

| プリセット | 打鍵化できる印字文字 |
|---|---|
| `en_us` | ASCII 印字文字(U+0020〜U+007E) |
| `ja_jp` | ASCII 印字文字(U+0020〜U+007E)+ `¥`(U+00A5) |

読み捨てになった文字も含め、消費された文字はすべて文字対応の出力(`writeText`)には届きます(打鍵化の可否は HID 出力側だけの制約)。

制御文字の標準割り当て:

| 文字 | 打鍵 |
|---|---|
| 0x09 (TAB) | Tab |
| 0x0A (LF) | Enter |
| 0x0D (CR) | Enter(直後の LF は読み捨て。CRLF → Enter 1 回) |
| 0x08 (BS) | Backspace |
| 0x1B (ESC) | Escape |
| その他の制御文字 | 読み捨て + エラーカウント |

### キー操作指示(アクション列)

**合成された打鍵指示(down / up / delay)の順序付き列**です。ユーザーの実キー押下は常に継続系であり、この列を経由しません。

発生源:

- キーシーケンスマクロ(Ctrl+C → Alt+Tab → Ctrl+V、途中の delay)
- 文字ストリームの展開結果(出力の縁で合流)
- 合成タップ(順序保証が必要な文脈のもの)

規則:

- 指示は消化時に**出力の押下集合へ順序どおり作用**します(単発系と継続系の接続点)。
- **文字 1 個分の枠(Shift down → キー down/up → Shift up)はアトミック**で、間に他のイベントを挟みません。
- ユーザーが押している修飾キーとは「文字ごとに退避 → 打鍵 → 復元」で共存します。オプションで「修飾が解放されるまで消化を遅延」を持ちます(既定: 到着順で即時)。
- ペーシング(report 間隔、文字間ディレイ)はアダプタ側。キューは固定容量で、空きは照会できます(背圧)。文字列投入はキューが埋まった時点で**捨てずに停止**し、消費量を返して呼び出し側が続きから再開できます。再開できない経路(マクロ内部、1 文字投入)だけ、あふれたら捨ててエラーカウントします。
- 実装ノート: 修飾変更と打鍵は別 report に分けて送る(同一 report での解釈順の揺れ対策)。

### 相対値

`(軸, 変化量)` のイベントです。軸は X / Y / ホイール / 横ホイール(マウス系)。

- **加算可能**: 同一軸の連続イベントは合算で圧縮できます(キュー詰まり時は文字と違い捨てずに縮められる)。
- 出力は report 間隔ごとに合算し、report の値域を超えた分は次回へ繰り越します。
- 軸ごとの**反転と整数倍率**を core の変換として持ちます(ナチュラルスクロール等)。ポインタ加速のような曲線変換は対象外。

## Lock 状態と LED

### 正本の規則

- lock を解釈するのはホストです。**lock を報告できる出力(USB device / BLE HID)が存在する間は、そこが正本**です。ブリッジは中継のみで、自前 toggle はしません。
- **正本 = アクティブな lock 報告出力(常に 1 つ)**。v1 では出力切替機能がないため「最初に登録された lock 報告出力」で固定します。複数の lock 報告出力が登録された場合、他ホストの lock 状態のズレは関知しません(禁止はしないが整合保証もしない)。lock 報告出力は 1 つを推奨します。
- **正本が存在しない間(USB 未接続、出力が UART 等のみ)は、ブリッジが終端ホスト**として正本を持ち、lock キーの press で内部状態を toggle します(対象: CapsLock / NumLock / ScrollLock。カナは対象外)。外部正本が現れたら最初の report で上書き(外部の勝ち)、消えたら最終値から引き継ぎます。

### 内部シャドウと通知

- ブリッジは内部に LockState を常に 1 つ保持します(起動時全 off)。正本からの report で上書きされ、変化があれば**全入力アダプタへ通知**します。
- 通知の実現手段はアダプタ依存です(USB キーボードへは LED report、GPIO 自作キーボードへはピン出力、BLE へは output report)。
- 後から接続された入力には、接続時点の内部状態を通知します(ホストは変化時しか report を送らないため、内部保持が必須)。
- 能動 lock 同期(ズレたホストへ toggle キーを注入して合わせる)は行いません(将来メモ)。

## キーボードのレイアウト変換

物理キーボードの刻印 layout とホスト OS の layout 設定が異なる構成(例: US 刻印キーボードを日本語 Windows で使う)のための変換です。入力ごとに opt-in します。

### 変換の構造

- 必要なのは **(キー, Shift 状態) → (キー, Shift 状態)** の変換表です。usage 単体の置換では Shift 面の記号を表現できません(US の Shift+2=`@` は ja_jp では 0x2F 単打)。
- 英字・数字(単打)・ナビゲーション・修飾キー等の**恒等キーは素通し**します。変換対象は記号系のみ(US⇄JA で約 15 キー)。
- **Shift の抑制と合成**: 物理 Shift があっても出力で外す、なくても出力で押す。押している間は維持する(長押し・リピートを保つ)。
- 物理キーボードに存在しないキーも出力は送れます(`_` = Shift+0x87 等)。変換/無変換/かな などの IME キーの供給は変換表ではなく**通常の remap** で行います。
- キーボードレイアウト記述(文字⇄(キー, Shift))から変換表を導出するか、ペア表を直接持つかは実装判断です。プリセットの命名は locale 形式(`en_us`、`ja_jp`)とします。
- v1 の対象は Shift 面までです(AltGr を使う layout は将来メモ)。

### 運用要件と制約

- **変換の on/off 切替(ホットキーまたは外部制御)を必須機能とします。** 変換は「ホストが特定 layout で解釈する」前提の出力を作るため、BIOS / UEFI / リカバリ環境など別解釈の場面では逆に化けます。
- 修飾コンボ(Ctrl / Alt / GUI 押下中)のキーは既定で変換せず素通しします(位置ベースのショートカット優先。設定で変換に切替可)。英字ショートカットは恒等なのでどちらでも無傷です。
- Shift 要求が矛盾するキーの同時押し(`@` と `"` を同時に保持等)は表現不能で、非対応と明記します。
- 複数キーボードのレイアウト混在は「入力ごとの変換(統合前)+ 和集合統合」で成立します。個別に PC へ挿す方式では原理的に解決できません(OS layout は単一、OS の修飾はグローバル)。制約は「一方の修飾押下中に他方の変換キーが同時到来した瞬間」の干渉のみで、直列化と遅延オプションで緩和します。
- 機械タイピスト(HID 型バーコードリーダー等)は同じ変換表で被覆でき、ショートカット・同時押しの制約が実質発生しない最良条件です。
- 方向による実用度差があります: US 刻印 → ja_jp ホストは ASCII 被覆 100%。ja_jp 刻印 → en_us ホストは `¥` 等が打てず被覆に欠けが出ます。

### 文字列打鍵との関係

文字ストリームの展開(文字 → 打鍵)は、同じキーボードレイアウト記述を使います。数字は keypad ではなく主キー列へ展開し(NumLock 非依存)、英字は内部シャドウの CapsLock を参照して Shift を補正します。表にない文字(かな漢字、対象レイアウトにない記号)は読み捨ててエラーカウントします。

## 出力の規則

- 出力アダプタは**自分が表現できるものだけを出し、他は黙って捨てます**。マウス report はマウス対応出力を登録した場合のみ(USB Device 出力は常に複合 = 常にマウス対応)、ログ系出力は virtual キーや文字も出力可。
- virtual キーはパイプライン内で消費されるのが原則で、出力アダプタへは出ません(ログ系の診断出力を除く)。
- 出力は**状態スナップショット方式**です(USB HID report と同じ考え方)。boot keyboard の 6KRO 制限は出力アダプタ側の制約で、中間の押下集合には及びません。
- **文字ストリーム(`writeText`)の源は 2 つ**: `typeText()`/`typeChar()`/マクロ由来の文字と、`convertLayout()` を設定した入力の押下由来の文字(刻印でデコード、押下エッジで 1 文字、Enter→改行 / Tab→タブ)。文字対応出力(テキスト UART・ログ等)はこれを受け、HID 出力は無視する。これにより「HID キーボード/バーコードの押下」を文字として取り出せる(レイアウトは入力側 convertLayout の 1 箇所で決まる)。
- USB device の構成は descriptor で enumerate 時に確定します(USB Device 出力は常に keyboard + consumer + mouse の複合。未使用 interface は report を送らないだけで無害)。実行中の構成変更は再接続を要します。

## 原理的な不可能性(仕様で解決しないこと)

| # | 内容 |
|---|---|
| 1 | **打てない文字**: 出力先 layout に存在しない文字は HID では生成不能(en_us ホストへの `¥` 等) |
| 2 | **ショートカット意味論の判別**: アプリのキー束縛が文字ベースか位置ベースかはブリッジから観測不能。レイアウト変換はどちらに倒しても壊れるケースが残る |
| 3 | **修飾状態はホストに 1 つ**: 矛盾する Shift 要求の同時押しは表現不能。打鍵列とユーザー修飾は時間的に排他 |
| 4 | **ホストの見えない状態**: IME のオン/オフ、フォーカス中のアプリは観測不能。文字系機能は IME オフ前提(IME 化けはブリッジで解決不能。緩和は前置キー等のオプションのみ) |
| 5 | **物理 Fn キー**: キーボード内部で処理され HID に現れないため、観測・remap・共有とも不可能 |
